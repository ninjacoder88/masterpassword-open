# Executive Summary

This application implements a master password-based vault system for securely storing user credentials and notes, leveraging custom authentication and encryption mechanisms. Its core assets include user session tokens, encrypted vault data, and cryptographic keys used for access control and data protection. The system’s trust boundaries separate unauthenticated users (API access without valid tokens) from authenticated users (with valid Bearer tokens), while administrative functions are implicitly restricted through authorization enforcement. Key risks stem from weak cryptographic practices involving predictable pseudo-random number generators (PRNGs) for tokens and passwords, insecure key derivation using base64 truncation instead of a proper KDF, and potentially unauthorized cross-user note creation due to insufficient validation. These issues expose potential privilege escalation, credential theft, and denial-of-service through server-side request forgery (SSRF).

# Prioritised Recommendations

## 1. **Critical: Replace `System.Random` with Secure RNG**
- **Where:** `MasterPassword.BusinessLogic/TokenGenerator.cs`
- **What:** Replace `new Random()` with `RandomNumberGenerator` or `CryptoServiceProvider`.
- **STRIDE/A02:** Spoofing, Cryptographic Failures
- **Impact:** Prevents token prediction, mitigates impersonation.

## 2. **Critical: Implement Proper Key Derivation Using PBKDF2**
- **Where:** `MasterPassword.BusinessLogic/KeyDeriver.cs`
- **What:** Replace base64 truncation with `Rfc2898DeriveBytes` using at least 100,000 iterations.
- **STRIDE/A02:** Tampering, Information Disclosure
- **Impact:** Mitigates key derivation weakness, protects against brute-force/key recovery.

## 3. **Critical: Enforce Cross-User Ownership Checks on Notes**
- **Where:** `CreateNoteRequestHandler.cs`, `UpdateNoteRequestHandler.cs`
- **What:** Validate that all note operations are tied to the correct primary account ID.
- **STRIDE/A01:** Elevation of Privilege
- **Impact:** Prevents unauthorized note creation across accounts.

## 4. **High: Remove `UseDeveloperExceptionPage()` from Production**
- **Where:** `Startup.Configure(IApplicationBuilder)` in `Startup.cs`
- **What:** Conditionally apply only in development environments.
- **STRIDE/A05:** Information Disclosure
- **Impact:** Prevents leakage of internal stack traces and configuration info.

## 5. **High: Add Authenticated Encryption (AEAD) to Data Handling**
- **Where:** `Encryptor.cs`
- **What:** Switch to AES-GCM or ChaCha20-Poly1305 for authenticated encryption.
- **STRIDE/A02:** Tampering, Information Disclosure
- **Impact:** Ensures both confidentiality and integrity of sensitive data.

## 6. **Medium: Implement CSRF Protection for API Endpoints**
- **Where:** All API controllers using `[ApiController]` or manual form submission.
- **What:** Enable antiforgery token validation or enforce `SameSite=Strict` cookies.
- **STRIDE/A03:** Tampering
- **Impact:** Reduces risk of cross-site request forgery.

## 7. **Medium: Add Audit Logging for Sensitive Operations**
- **Where:** Note creation/modification handlers, password generation.
- **What:** Log successful and failed access attempts, password regenerations, etc.
- **STRIDE/A09:** Repudiation
- **Impact:** Enables forensics and compliance monitoring.

## Architectural Improvements:
- **Rate-limit healthcheck endpoints** to prevent DoS amplification.
- **Centralize authentication middleware** instead of relying solely on `[Authorize]` attributes.
- **Enforce strict CORS policies** if needed, especially for authenticated zones.
- **Move secrets to environment variables or Azure Key Vault** and remove them from config files.
